## Root Cause(s) Found

### 1. PRIMARY: `_colony/` mutable state is tracked in git (CRITICAL)

**80 files** in `_colony/` are git-tracked, including:
- `_colony/queue/*.md` — task queue files
- `_colony/active/*.md` — active task files
- `_colony/locks/*.lock` — lock files

**Mechanism:** When Agent A claims TASK-002 (`mv queue/ → active/`), the filesystem state is correct. But Agent B's earlier commit still has `queue/TASK-002.md` in git history. When Agent C does `git checkout` or `git pull`, git restores the old queue file. Result: **TASK-002 appears in queue/, active/, AND review/ simultaneously.**

This is why 6 tasks exist in multiple stages:
| Task | Locations |
|---|---|
| TASK-002 | queue + active + review |
| TASK-003 | active + review |
| TASK-005 | queue + active + review |
| TASK-006 | queue + active |
| TASK-007 | queue + active |
| TASK-009 | queue + active |

**Fix:** `.gitignore` mutable `_colony/` subdirectories. Remove them from git tracking.

### 2. SECONDARY: GitHub default branch is `task/TASK-001` instead of `main`

```
$ gh repo view TomOst-Sec/J-R --json defaultBranchRef
{"defaultBranchRef":{"name":"task/TASK-001"}}
```

`origin/HEAD` resolves to `task/TASK-001`. Agents pushing to `origin main` push to a non-default branch. `git clone` would checkout the wrong branch.

**Fix:** `gh repo edit TomOst-Sec/J-R --default-branch main`

### 3. TERTIARY: Shared working directory — only 3 worktrees for 10 agents

**Worktrees:**
```
/home/tom/.atlas/projects/J-R  → task/TASK-017 (shared by most agents)
/tmp/task-001-wt               → task/TASK-001
/tmp/task-008-wt               → task/TASK-008
```

The main repo working directory is on `task/TASK-017`, not `main`. When audit does `git checkout main`, it switches the branch for every agent sharing that directory. Agents trip over each other's branch checkouts, causing:
- `error: you need to resolve your current index first`
- `error: cannot pull with rebase: You have unstaged changes`
- Stash/pop conflicts

alpha-2 is correctly using worktrees (`/tmp/alpha2-wt-011`). Most others are not.

### 4. QUATERNARY: Pre-commit hook blocks legitimate "move to review" commits

bravo-1 tried to commit a review-move to main and got blocked:
```

The pre-commit hook prevents coders from committing `_colony/` state changes to main, even though pipeline state transitions (claim/complete) MUST update files on the shared filesystem. This creates a deadlock: coders need to move files but can't commit the moves.

Long sleep cycles mean these agents can't respond to the deadlock. CEO and atlas issued cycle 1 reports based on stale data and went to sleep. beta-tester only tested TASK-001 scaffold.

- If in active/ AND queue/ → remove from queue/

```
